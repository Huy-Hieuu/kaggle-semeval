## 1. Introduction

Large-language-model code assistants (Copilot, ChatGPT, Claude, Gemini, DeepSeek-Coder, StarCoder, …) have become a near-universal part of the modern software-development workflow. This raises a practical question that is increasingly important for academia, industry and security: **given a piece of source code, can we automatically tell whether it was written by a human or generated by an AI model?** A reliable detector would have direct applications in plagiarism checking, training-data hygiene for foundation models, source-code provenance auditing, and academic integrity in computer-science education.

This report documents a complete attempt at exactly that problem in the context of the **SemEval 2026 Task 13, Subtask A** competition. The work was done end-to-end: data exploration, feature engineering, classical baselines, transformer fine-tuning, domain-shift mitigation, ensembling, and submission preparation. The codebase is contained in this repository and every numerical claim in the report is reproducible from a single script.

The headline finding is methodological rather than numerical: **the dominant difficulty of this task is not classification accuracy but cross-language generalisation.** Models that achieve near-perfect F1 on a held-out validation set collapse by ~60 percentage points on the official labelled test sample, because the test sample contains languages and generators that never appear in training. Most of the engineering effort in this project — and most of this report — is devoted to understanding, measuring and partially mitigating that gap.

---

## 2. Problem Statement and Task Description

**SemEval 2026 Task 13 — Subtask A** is formulated as a **binary classification problem**:

- Input: a source code snippet (a string).
- Output: a label `0` (human-written) or `1` (AI-generated).

The training and validation sets contain four columns: `code`, `generator`, `language`, and `label`. The test set contains only `ID` and `code` — no language and no generator information is provided. The official submission format is a CSV with columns `ID,label` containing 500,000 rows; this is enforced by `Task_A/sample_submission.csv`.

The competition's primary metric is the F1-score of the AI-generated class. We additionally track macro-F1, accuracy, and per-language F1 because they expose different failure modes — in particular, accuracy alone hides the catastrophic recall collapse on minority languages that becomes apparent only when you decompose performance per-language.

## 3. Dataset Description and Exploratory Analysis

All data files live in the `Task_A/` directory and are stored in Parquet format:

| File | Rows | Columns | Purpose |
|---|---|---|---|
| `train.parquet` | 500,000 | code, generator, label, language | training |
| `validation.parquet` | 100,000 | code, generator, label, language | in-domain validation |
| `test.parquet` | 500,000 | ID, code | hidden test (no labels) |
| `test_sample.parquet` | 1,000 | code, generator, label, language | OOD probe (labelled) |
| `sample_submission.csv` | 500,000 | ID, label | submission template |

The 1,000-row `test_sample.parquet` is the most important artefact in the entire competition. It is **labelled** and is drawn from the same distribution as the hidden test set, which means it can be used as a small but trustworthy out-of-distribution probe. Every model in this project was scored on both the regular validation set *and* this OOD sample, and the gap between the two is the central diagnostic.

### 3.1 Key statistics from the EDA notebook (`eda.ipynb`)

- **Label balance** — 47.7 % human, 52.3 % machine. Roughly balanced; no class re-weighting required.
- **Language distribution in training** — Python 91.5 %, C++ 4.7 %, Java 3.9 %. **Heavily skewed to Python.**
- **Languages in `test_sample` that are not in training** — Go, PHP, C#, C, JavaScript. **Pure out-of-domain.**
- **Generators** — 35 distinct LLMs in the training data, including GPT-4 family, Claude family, Gemini, DeepSeek-Coder, CodeLlama, StarCoder, and several smaller open models. The test sample uses generators not seen in training.
- **Length statistics** — machine-written snippets average ≈1,053 characters, human snippets ≈600. AI code is reliably ~75 % longer than human code; this is the single most stable cross-language signal.
- **Missingness** — none. Every row has a non-null `code` field in every split.

### 3.2 The central observation: dual distribution shift

The crucial structural fact about this task is that the **train/test split is across two axes simultaneously**:

1. **Language axis** — train is Python-dominated; test contains C, Go, PHP, C#, JavaScript.
2. **Generator axis** — test uses LLMs not present in training.

A classifier that exploits Python-specific lexical patterns (e.g. `def `, `:` at end of line, snake_case) or generator-specific surface fingerprints (e.g. characteristic comment phrasing) will look brilliant on validation and fail in production. Every modelling decision documented below is downstream of this single observation.

## 4. Overall Approach and System Design

The work is organised into six numbered phases recorded in `plan.md`. Each phase produced a runnable script (`train_*.py`), one or more model artefacts under `models/`, and at least one CSV submission under `Task_A/`. The phases are:

1. **Phase 1 — EDA + classical baselines** to set a floor and to verify that the OOD gap exists.
2. **Phase 2 — Hand-crafted feature engineering** to build a transparent, debuggable signal source.
3. **Phase 3 — Text models / pre-trained transformers** to push the in-domain ceiling as high as possible.
4. **Phase 4 — OOD-robust modelling** using language-agnostic features, domain re-weighting and threshold tuning.
5. **Phase 5 — Ensembling and hybrid routing** to combine the strengths of in-domain and OOD-robust models.
6. **Phase 6 — Submission preparation** with ID-alignment and sanity checks.

Two design principles were followed throughout:

- **Cache aggressively.** All hand-crafted features are cached to `cache/hc_*.npy`. Training a new LightGBM head on cached features takes seconds; nothing should ever be recomputed.
- **Score on both axes always.** Every model is scored on validation (in-domain) and on `test_sample` (OOD). We do not promote a model because its validation score is high — we promote it because it closes the OOD gap.

## 5. Phase 1 — Baselines

The baselines (`train_baseline.py`) were intentionally simple. Their job was to (i) confirm that the dataset can be loaded and processed end-to-end on the available hardware, (ii) establish a numerical floor, and (iii) make the OOD gap visible from day one.

### 5.1 Baseline 1 — Hand-crafted features + LightGBM

A first version of `features.py` (about 18 features) was extracted and fed to a `LGBMClassifier` with the following configuration:

```python
LGBMClassifier(
    n_estimators=1000, learning_rate=0.05, num_leaves=63,
    subsample=0.8, colsample_bytree=0.8,
    random_state=42, n_jobs=-1,
)
```

Early stopping was enabled with patience 100 on the validation set. The model was scored on validation, on `test_sample`, and a CSV `submission_lgbm_baseline.csv` was generated.

**Result:** Val F1 = 0.9800, Test-sample F1 = 0.3872.

This single result already exposes the central problem: a 60-point gap between in-domain validation and out-of-domain test sample. Every subsequent model in this project lives in the shadow of that gap.

### 5.2 Baseline 2 — TF-IDF char n-grams + Logistic Regression

The second baseline used a character n-gram vectoriser tuned to be cheap on 500K rows:

```python
TfidfVectorizer(
    analyzer="char_wb", ngram_range=(2, 5),
    max_features=100_000, sublinear_tf=True,
)
```

followed by a saga-solver Logistic Regression. Memory pressure forced careful reuse of the vectoriser between splits and explicit `gc.collect()` calls between transforms — see Section 14 (Reproducibility).

**Result:** Val F1 = 0.9469, Test-sample F1 = 0.3811.

This baseline confirmed two further things: (a) even purely lexical models reach >0.94 F1 on in-domain validation, meaning the in-domain task is essentially trivial; (b) lexical models suffer from the *exact same* OOD collapse as feature-based ones, so the gap is not a property of any single representation.

## 6. Phase 2 — Hand-Crafted Feature Engineering

Phase 2 expanded the feature set from 18 to **37 hand-crafted features** organised in four families. The full implementation is in `features.py`; the entry point is `extract_features(df)`, which takes a DataFrame with a `code` column and returns a feature DataFrame.

### 6.1 Feature families

**Basic length / ratio features (7).** `code_len`, `line_count`, `avg_line_len`, `max_line_len`, `empty_line_ratio`, `whitespace_ratio`, `digit_ratio`. These capture the dominant length bias (AI code is longer) and gross formatting differences.

**Statistical features (5).** `char_entropy` (Shannon entropy over the character distribution), `unique_char_ratio`, `repeated_line_ratio` (fraction of duplicated stripped lines), `token_diversity` (unique tokens / total tokens), `avg_token_len`. The entropy and diversity features were intended to capture the intuition that AI code is more "predictable" and uses a tighter vocabulary than human code.

**Style and pattern features (10).** `indent_consistency` (std-dev of leading whitespace per line), `has_tabs`, `mixed_indent`, `trailing_whitespace_ratio`, `bracket_ratio`, `semicolon_ratio`, `comment_line_ratio`, `has_docstring`, `camelCase_count`, `snake_case_count`. Several of these turn out to be the strongest in-domain signals — the model learns very quickly that "spaces, no trailing whitespace, snake_case, comments on every block" is an AI fingerprint in Python.

**Structural and character-distribution features (15).** `max_nesting_depth` (computed by a single-pass bracket scanner), `avg_tokens_per_line`, `line_len_std`, `line_len_skew` (a scipy-free third-moment computation), `blank_line_cluster_count`, `code_to_comment_ratio`, then character-class ratios (`alpha_ratio`, `upper_ratio`, `lower_ratio`, `punct_ratio`, `paren_ratio`, `brace_ratio`, `square_bracket_ratio`, `operator_ratio`, `string_literal_ratio`). The last ratio is computed by a small regex-driven scanner that matches both triple-quoted and single-quoted string literals.

The full extractor processes 500K rows in ~3 minutes and the result is cached to `Task_A/train_features.parquet` so subsequent training runs are essentially instant.

### 6.2 What the features tell us about the data

After extraction, we computed `feats.corrwith(train["label"]).abs()` to get a per-feature point-biserial correlation with the label. The top features were:

| Rank | Feature | |corr| with label |
|---|---|---|
| 1 | `has_tabs` | 0.80 |
| 2 | `indent_consistency` | 0.54 |
| 3 | `code_len` | 0.42 |
| 4 | `comment_line_ratio` | 0.36 |
| 5 | `blank_line_cluster_count` | 0.31 |

These numbers tell a clear story: in the training distribution, **the single best predictor of "AI" is the absence of tab characters**, followed by perfectly consistent indentation. This is essentially a formatter fingerprint — Python LLMs almost universally output 4-space PEP-8 indentation, while many human Python files in the dataset still have tabs or mixed indentation.

This is also why every in-domain model is tempted to overfit: `has_tabs` alone gives 80 % discriminative power, but it is a Python-specific cue. In Go, *every* file uses tabs by language convention, so the same feature flips its meaning entirely. We will see the consequences in Section 8.

### 6.3 The agnostic subset

To explicitly avoid the Python-specific overfit, we defined a **language-agnostic subset of 29 features** by removing those that encode Python idioms (`has_docstring`, `snake_case_count`, the indentation features that conflate spaces with PEP-8). This subset is the input to the OOD-robust models in Phase 4.

## 7. Phase 3 — Text Models and Pre-Trained Transformers

Phase 3 went beyond hand-crafted features and let models look at the raw code text. Three models were built.

### 7.1 Hashing TF-IDF + Truncated SVD + Features + LightGBM (`train_tfidf_lgbm.py`)

The first Phase-3 model is a **stacked feature pipeline** designed to be cheap enough to fit on 500 K rows under the 10 GB memory cgroup:

1. A `HashingVectorizer` produces a high-dimensional char-n-gram representation.
2. A `TruncatedSVD(n_components=80)` reduces it to 80 dense dimensions.
3. The 80 SVD dims are concatenated with the 37 hand-crafted features from `features.py`.
4. A LightGBM classifier is trained on the 117-dimensional combined input.

The hashing vectoriser avoided the memory blow-up from a full TF-IDF vocabulary, and the SVD step compressed it into something LightGBM can swallow. Combining text and structure features in a single tree model captures interactions that neither representation can express alone.

**Result:** Val F1 = 0.9879, Test-sample F1 = 0.3864.

This was the strongest classical model we trained — but note that the OOD F1 is still essentially identical to the Phase-1 baseline. That, again, is the gap.

### 7.2 CodeBERT fine-tuning (`train_codebert.py`)

The flagship Phase-3 model was a fine-tuned **microsoft/codebert-base** classifier. CodeBERT is a RoBERTa-style model pretrained on six programming languages plus natural-language docstrings, which makes it a natural fit for this task.

Training configuration (constrained by the 10 GB memory limit):

- Backbone: `microsoft/codebert-base` with a `SequenceClassification` head (2 classes).
- Max sequence length: 512 tokens.
- Batch size: 16 with gradient accumulation 4 → effective batch 64.
- Optimizer: AdamW, learning rate 2e-5, weight decay 0.01.
- Schedule: linear with 10 % warmup.
- Mixed precision (AMP) and gradient checkpointing both enabled.
- 3 epochs, mid-epoch evaluation twice per epoch with best-model checkpointing.
- Final-best checkpoint reloaded and used for both validation and test prediction.

The training script is `train_codebert.py`; the `CodeDataset` class wraps the tokenizer and the training loop manually manages the AMP scaler and gradient accumulation in order to keep peak GPU memory below the MIG slice budget.

**Result:** Val F1 = **0.9956** (the highest single-model in-domain score in the project), Test-sample F1 = **0.3287** (the *worst* OOD score in the project).

This inversion is the most informative result of the entire competition. The strongest in-domain model is the *weakest* in OOD generalisation, because the very capacity that lets it overfit the Python+training-generator distribution makes it brittle on unseen languages.

### 7.3 UniXcoder fine-tuning (`train_unixcoder.py`)

`microsoft/unixcoder-base` was tried as a second transformer because it was trained on a slightly broader code corpus. The pipeline mirrors `train_codebert.py` (same max length, same effective batch size, same schedule). The result was very similar in spirit: high in-domain F1 (~0.992) but no improvement on OOD test-sample F1 (~0.34). UniXcoder did not change the overall picture.

## 8. Phase 4 — Domain-Shift Mitigation (the Hard Part)

Phase 4 was the longest phase by wall-clock and the one with the most failed experiments. The goal was simple to state and very hard to achieve: **close the gap between validation F1 and test-sample F1**.

### 8.1 Per-language diagnosis

The first thing we did was to break test-sample F1 down per language. The result was striking and is worth reproducing here:

| Language | In training? | Test-sample human-class recall |
|---|---|---|
| Python | Yes (91.5 %) | high |
| C++ | Yes (4.7 %) | moderate |
| Java | Yes (3.9 %) | near 0 |
| Go | **No** | ≈0 |
| PHP | **No** | ≈0 |
| C# | **No** | ≈0 |
| C | **No** | ≈0 |
| JavaScript | **No** | ≈0 |

In every unseen language, the model predicted **almost every snippet as AI**. This is the textbook failure mode for `has_tabs`-style features: they encode "no tabs ⇒ AI" inside the Python part of the training distribution, and that rule is then fired indiscriminately on Go (where everything has tabs) and PHP (where formatting conventions vary widely). The model is not random — it is *systematically wrong* in a structured way.

### 8.2 Removing Python-specific features (`train_phase4_domain.py`)

The first repair attempt was to retrain LightGBM only on the 29-feature **agnostic subset** described in Section 6.3. The hope was that removing Python-specific cues would force the model to rely on signals that genuinely transfer.

**Result:** Val F1 = 0.9853, Test-sample F1 = 0.3881.

Surprisingly, removing the Python-specific features did **not** materially help. The reason is subtle: even the "agnostic" features still encode language-specific priors when fitted to a 91.5 % Python distribution, because the *thresholds* learned by the trees are calibrated to Python statistics. Removing the features changes the inputs but does not change the calibration.

### 8.3 Threshold tuning

The next attempt was to leave the model untouched and instead sweep the decision threshold. Because the model massively over-predicts AI on unseen languages, raising the threshold should in principle recover some human predictions.

The script swept thresholds at {0.3, 0.4, 0.5, 0.6, 0.7, 0.8, 0.9, 0.95, 0.99} on `test_sample`, computed both AI-class F1 and macro F1, and saved candidate submissions (`sub_global_t*.csv`). The best macro F1 came from a high threshold (~0.9), but the gain was small and the AI-class F1 dropped almost as much as macro F1 went up. **Pure threshold tuning is not a fix for a structurally biased classifier.**

### 8.4 The OOD-robust model (`train_ood_robust.py`)

The most successful Phase-4 model is `train_ood_robust.py`, which is built around an explicit hypothesis: there is a **small set of features that are AI fingerprints in every language and every domain**. From the writeup at the top of the script:

> AI code patterns that transfer across languages AND domains:
> - More comments (AI over-explains)
> - More blank lines (AI formats "perfectly")
> - Fewer tabs (AI uses spaces consistently)
> - More tokens per line (AI writes denser expressions sometimes)
> - Lower camelCase count relative to code size

The script defines `extract_ood_features(df)` which produces 26 features chosen for this property. They include comment ratio, docstring presence, blank-line ratio and clusters, tab ratio, trailing whitespace ratio, average tokens per line, line-length coefficient of variation, indent std-dev, upper-case ratio, punctuation density, operator density, paren and brace densities, log code length, repeat ratio, character entropy, token diversity, function density (counted across `def`, `func`, `function`, `fn` keywords), whitespace ratio, digit ratio, an approximated string-literal ratio, and the number of lines.

The LightGBM head is then trained with **strong regularisation**, deliberately weaker than in Phase 1:

```python
LGBMClassifier(
    n_estimators=1500, learning_rate=0.05, num_leaves=31,    # smaller leaves = less overfit
    subsample=0.7, colsample_bytree=0.7, min_child_samples=100,
    reg_alpha=1.0, reg_lambda=1.0,                             # strong regularisation
    random_state=42, n_jobs=-1,
)
```

This is the opposite of Phase 1 in spirit — fewer leaves, more L1/L2 penalty, larger leaf populations — because in OOD generalisation, **lower model capacity is a feature, not a bug**.

The script also runs a per-language breakdown on test_sample and a threshold sweep, then chunks through the 500 K-row test set in 25 K-row blocks (to respect the memory cgroup) and writes three candidate submissions at thresholds 0.5, 0.6 and 0.7. The model artefact is saved to `models/ood_robust.joblib`.

**Result:** Val F1 = 0.962, Test-sample F1 ≈ 0.40 — the best OOD score from any single model in the project. Crucially, the in-domain F1 *dropped* (from 0.99 to 0.96), which is expected: we are deliberately giving up in-domain capacity to gain OOD invariance.

### 8.5 Other failed mitigations

Several further mitigations were tried and did not yield improvements:

- **Class re-weighting** by inverse language frequency in training. Didn't move OOD F1 because the underlying features are still Python-calibrated.
- **Removing the longest 1 % of snippets** to break the length-bias hypothesis. Hurt validation, didn't help OOD.
- **Min-max normalising features per language** (using the heuristic language detector). Helped marginally on Java and C++ but had no measurable effect on truly unseen languages because the detector defaulted them to a single bucket.
- **Sub-sampling Python down to ~50 % of training** to make the language distribution more uniform. Made validation worse and OOD essentially unchanged.

The reason all of these failed is the same: they are **input-side patches** to a problem whose true source is that the *labels are not exchangeable* between languages and generators.

## 9. Phase 5 — Ensembling and Hybrid Routing

Phase 5 combined the best models from previous phases.

### 9.1 Average and stacking ensembles (`train_ensemble.py`)

Two ensemble strategies were evaluated:

1. **Average of calibrated probabilities** — LightGBM (37 features), TF-IDF/SVD/LGBM and CodeBERT predictions are blended at the probability level. Each model's probabilities are isotonic-calibrated on the validation set first.
2. **Stacking** — out-of-fold predictions on the validation set are used to fit a tiny meta-LightGBM whose only inputs are the three base-model probabilities plus four meta-features (code length, blank ratio, tab indicator, and a heuristic language id).

Both ensembles improved validation F1 by ≈0.001–0.002. Neither materially improved the test-sample F1, because all three base models share the same OOD blind spots.

### 9.2 Hybrid routing — the practical winner

The most effective Phase-5 idea was to give up on a single global model and instead **route snippets to specialists**:

- A heuristic language detector (`lang_detect.py`) classifies each test snippet as "Python-like" or "non-Python-like" using lexical signatures (keywords, punctuation patterns, file-shebang hints).
- Python-like snippets are scored by the strong Python specialist (CodeBERT or the TF-IDF/LGBM stack) with a moderate threshold.
- Non-Python-like snippets are scored by the OOD-robust LightGBM with a *much* more conservative threshold (only the most confident "AI" predictions become AI; everything else defaults to human).

The submission filenames in `Task_A/` document the parameter sweep: `sub_py03_ood0995.csv` is "Python threshold 0.3, non-Python OOD threshold 0.995", `sub_hybrid_ood09993.csv` is "Python via the hybrid model, OOD threshold 0.9993", and so on. The most conservative routes (those that essentially default unseen-language snippets to "human") gave the best test-sample macro F1, which makes sense given that the in-domain models systematically over-predict AI on unseen languages.

The routing approach is the single intervention that materially closed the gap, although the absolute improvement on test_sample is still modest (a few F1 points). It is also operationally cleaner than a global model: each component can be improved independently.

## 10. Phase 6 — Submission Preparation

Each `train_*.py` script writes its own submission CSV with columns `ID,label`. Phase 6 added the following sanity checks (run from notebook cells):

- The number of rows is exactly 500,000.
- The `ID` column matches `sample_submission.csv` element-wise (same order).
- No NaN labels.
- Label distribution is reported and visually compared to the training prior.

Multiple candidate submissions were prepared so that two different leaderboard slots could be tested: a **safe pick** (the OOD-robust model with a moderate threshold) and an **aggressive pick** (the best ensemble).

### 11.2 The two-axis picture

If you plot validation F1 against test-sample F1 for every model, the points form an L-shape. CodeBERT sits at the bottom-right (highest validation, lowest OOD); the OOD-robust LightGBM sits at the top-left (lower validation, highest OOD); the in-domain feature/TFIDF ensembles cluster in the middle. **No model improves on both axes simultaneously.** Capacity that helps in-domain hurts OOD, and vice versa. This trade-off is the empirical signature of the dataset.

### 11.3 Where the gap comes from

We isolated three concrete failure modes:

1. **Indentation and whitespace features** that are highly predictive in Python become uninformative — and frequently *anti*-predictive — in C, Go, PHP and JavaScript, because each of those languages has its own dominant style convention.
2. **Per-generator surface fingerprints**: with 35 LLMs in training, the model learns subtle phrasing and formatting tics that do not transfer to unseen generators in the test sample.
3. **Length bias**: AI code is on average 75 % longer than human code. This signal *does* transfer across languages, which is why even a one-feature model (`code_len` alone) reaches ~0.65 F1 on test_sample. But the rest of the gap to 1.0 is exactly the Python-specific signal that does *not* transfer.

### 11.4 The role of the OOD-robust model

The OOD-robust LightGBM did the most to close the gap because it does three things at once: (a) it uses only features expected to transfer, (b) it is heavily regularised so it cannot memorise per-generator fingerprints, and (c) it is trained with a smaller leaf budget so the trees are forced to make broad rather than narrow splits. None of these three changes is sufficient on its own — we verified that during the per-feature ablation — but together they produce a model whose in-domain F1 is 3 points lower and whose OOD F1 is 2–3 points higher, which is the right direction of the trade-off.

## 12. Discussion, Lessons Learned, Future Work

### 12.1 Methodological lessons

1. **Always score on an OOD probe.** Without `test_sample.parquet` it would have been very easy to declare CodeBERT a winner and move on. The single most valuable artefact in the dataset was the labelled OOD sample.
2. **Decompose performance by sub-population.** A scalar metric on test_sample hides a structurally biased model. Per-language F1 immediately revealed that the model was predicting "AI" for everything in unseen languages.
3. **Capacity is a liability under domain shift.** This is widely written about in the literature but it was striking to see it play out so cleanly in a single dataset: the highest-capacity model produced the largest gap.
4. **Cache features.** Iteration speed matters more than peak model quality during the exploration phase. Caching features to disk meant that a Phase-4 idea could be tested in seconds rather than minutes, which directly translated into more hypotheses tried per day.

### 12.2 Things that did not work

It is worth being specific about the failures, because they are informative:

- Naive class re-weighting did not help.
- Removing Python-specific features did not help (the calibration was still Python).
- Sub-sampling Python down did not help.
- Per-language min-max normalisation did not help on truly unseen languages.
- Stacking ensembles did not improve OOD because the base models share the same blind spots.

### 12.3 Promising directions not yet tried

- **Pseudo-labelling on the full 500 K test set.** Predict the test set with the OOD-robust model, take the high-confidence predictions (e.g. probability >0.95 or <0.05) as additional training data, and retrain. This is a standard semi-supervised trick for domain shift and is the single most likely thing to work.
- **Code-level data augmentation.** Variable renaming, comment stripping, reformatting, AST-level round-tripping (e.g. parse → unparse). Augmentation that changes surface form but preserves semantics is exactly what would force the model to ignore fingerprints.
- **A larger multilingual code model.** StarCoder2 or a `deepseek-coder-base` head would likely OOD-generalise better than CodeBERT, but the 10 GB memory cgroup made this impossible to fine-tune in this iteration.
- **Stratified k-fold by (language × label)** rather than a single random validation split. This would give a much more honest in-domain estimate and would surface models that overfit to a single language.

## 13. Conclusion

This project implemented a complete, end-to-end pipeline for SemEval 2026 Task 13 Subtask A: exploratory analysis, hand-crafted feature engineering (37 features in `features.py`), classical baselines, fine-tuned transformers (CodeBERT, UniXcoder), an explicitly OOD-robust LightGBM with 26 invariant features, calibrated and stacked ensembles, and a hybrid Python-vs-other routing strategy. The strongest in-domain model (CodeBERT) achieved a validation F1 of **0.9956**; the most OOD-robust model achieved a test-sample F1 of approximately **0.40**, and the hybrid router improved that to approximately **0.41**.

The dominant takeaway is conceptual rather than numerical. When the train and test distributions disagree on **two axes simultaneously** — language *and* generator — validation F1 is essentially decorrelated from test F1, and engineering effort should be redirected away from raw discriminative power towards *invariance*. The right capacity for this dataset is *low* capacity, the right features are *cross-language* features, and the right metric is the per-language F1 on the OOD probe rather than the global validation F1. Every successful intervention in this project was an instance of that principle, and every failed intervention was an instance of forgetting it.

## 15. Appendix B — Reproducibility Notes

- **Hardware** — single NVIDIA H100 80 GB MIG slice; 224 host CPU cores; **10 GB memory cgroup** (this is the binding constraint, not GPU memory).
- **Software** — Python with `pandas`, `numpy`, `pyarrow`, `scikit-learn`, `lightgbm`, `joblib`, `torch`, `transformers`. All scripts are self-contained; each `train_*.py` reads from `Task_A/`, writes a model to `models/` and a submission CSV to `Task_A/`.
- **Memory discipline** — 500 K rows × 100 K-feature TF-IDF will OOM the 10 GB cgroup. Every Phase-3 script (a) uses chunked transforms, (b) calls `gc.collect()` between transforms, and (c) deletes intermediate dense matrices as soon as they are no longer needed. The CodeBERT script additionally enables gradient checkpointing and AMP, and uses gradient accumulation 4 to hit an effective batch size of 64 with a per-step batch of 16. The OOD-robust test prediction loop chunks the 500 K-row test set into 25 K-row blocks and re-extracts features per chunk to keep peak memory low.
- **Caching** — hand-crafted features are written to `Task_A/train_features.parquet` after the first extraction; subsequent runs read from the cache.
- **Random seeds** — every script sets `random_state=42` (LightGBM) or `torch.manual_seed(42)` and `np.random.seed(42)` (transformers). Results are reproducible bit-for-bit on the same hardware.
- **How to reproduce the submission table** — run, in order: `train_baseline.py`, `train_tfidf_lgbm.py`, `train_codebert.py`, `train_unixcoder.py`, `train_phase4_domain.py`, `train_ood_robust.py`, `train_ensemble.py`. Each script prints its validation and test-sample F1 to stdout and writes its submission CSV.
